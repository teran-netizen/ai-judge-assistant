# Техническое задание: Аккумулятивный пайплайн обработки дел

**Проект:** AI Judge v3.13.0+
**Дата:** 2026-03-07
**Статус:** Черновик, требует проверки

---

## 1. Проблема

### Текущие ограничения
- `MAX_OCR_TEXT_CHARS = 120_000` символов (~30K токенов, ~48 страниц) — всё, что сверх, **молча обрезается**
- DeepSeek V3: контекст 64K токенов, из них ~30K на входной текст + ~8K на промпт + ~8K на генерацию = потолок ~48 страниц
- OCR rate limit: 1.1 сек между запросами (Yandex Vision API)
- Один вызов LLM на всё дело — нет структурированного извлечения

### Реальная потребность
- Крупные дела: 20–40 документов, 200–800 страниц
- Судья загружает фотографии документов пачками (по 1–20 штук за раз)
- Текущая система: обрезает после ~48 страниц, остальные документы **игнорируются без предупреждения**

### Целевой результат
- Обработка до ~200 фотографий документов
- Накопление контекста пачками: загрузил 20, потом ещё 20, потом генерация
- Качественный проект решения с ответом на все доводы сторон

## 2. Архитектура: Аккумулятор

### Три действия пользователя

```
Действие 1 → Судья загружает пачку фотографий (1–20 шт.)
              Система: OCR → извлечение → добавление в контекст дела
              Судья видит результат по каждому документу

Действие 2 → Судья загружает ещё фотографий
              Система: добавляет к накопленному контексту

Действие 3 → Судья нажимает «Сгенерировать решение»
              Система: берёт весь накопленный контекст → генерация → SSE-стрим
```

### Три модуля

```
[Модуль 1] Ingest    — OCR фото → извлечение → добавление в case_context
[Модуль 2] Context   — хранение и управление накопленным контекстом дела
[Модуль 3] Generate  — генерация решения из накопленного контекста
```

### Что убрано по сравнению с Map-Reduce

| Было (Map-Reduce) | Стало (Аккумулятор) | Почему |
|---|---|---|
| Роутер Short/Long Path | Один путь для всех дел | Меньше веток = меньше багов |
| Step 1 Detect (отдельный) | Тип документа определяется внутри Ingest | Не нужен отдельный шаг |
| Step 3 Reconcile (отдельный) | Сводка строится инкрементально | Каждый документ обновляет summary |
| Батчи по 5, параллельность | Последовательная обработка | OCR rate limit 1.1 сек, проще и надёжнее |
| Два пути кода | Один путь | Любой размер дела — один код |

## 3. Модуль 1: Ingest — обработка фотографий

### Поток обработки одной фотографии

```
Фото → OCR (Yandex Vision) → текст
    → проверка дубликатов (hash OCR-текста)
    → DeepSeek: извлечение структуры (с контекстом предыдущих документов)
    → результат добавляется в case_context
    → summary пересчитывается
```

### Последовательная обработка

Каждая фотография обрабатывается по очереди, не параллельно:
- OCR rate limit 1.1 сек — параллельность не даёт выигрыша
- Последовательность проще и надёжнее
- Каждый следующий документ обрабатывается **с контекстом предыдущих**

### Контекст при извлечении

DeepSeek получает краткий контекст уже обработанных документов:

```
Контекст дела (уже обработано 5 документов):
- Истец: Петров П.П.
- Ответчик: ООО «Стройинвест» (ИНН 5401234567)
- Спор: взыскание 1 500 000 руб. по договору подряда №15 от 01.03.2024
- Ответчик оспаривает качество работ

Теперь извлеки информацию из следующего документа:
[OCR-текст документа]
```

Это позволяет модели:
- Не дублировать уже известные данные о сторонах
- Правильно привязывать суммы/даты к контексту дела
- Определять роль документа в деле (подтверждает позицию истца или ответчика)

### Промпт извлечения

Задача промпта — **извлечь данные, не давать юридическую оценку**. Map = экстрактор, не судья.

```
Ты — помощник судьи. Извлеки из документа ВСЕ фактические данные.
Не давай юридическую оценку — только извлечение.

КОНТЕКСТ ДЕЛА:
{case_summary}

ДОКУМЕНТ:
{ocr_text}

ИЗВЛЕКИ:

1. ТИП ДОКУМЕНТА
   Определи: исковое, отзыв/возражение, договор, платёжный документ,
   экспертиза, протокол, справка/выписка, доверенность, переписка/претензия,
   расчёт, судебный акт, фото/скриншот, административный акт, иное.

2. РЕКВИЗИТЫ
   - Дата документа
   - Номер
   - Кто составил / выдал
   - Кто подписал, на каком основании (должность, доверенность — номер, дата)
   - Печати, штампы

3. СТОРОНЫ (если упоминаются новые или уточняются известные)
   - ФИО / наименование, ИНН, ОГРН, адрес
   - Представители: ФИО, реквизиты доверенности, объём полномочий
   - Третьи лица

4. СУММЫ И ДАТЫ
   - Каждая сумма: размер, валюта, за что, период
   - Каждая дата: что произошло
   - Расчёты: формула, ставка, период, результат

5. ДОВОДЫ И ПОЗИЦИИ
   - Что утверждает автор документа (ДОСЛОВНЫЕ формулировки для ключевых доводов)
   - Что признаёт
   - Что оспаривает
   - О чём молчит (по отношению к уже известным требованиям)

6. ССЫЛКИ НА НОРМЫ
   - Все упомянутые статьи законов, кодексов
   - Постановления Пленумов ВС РФ
   - Подзаконные акты (ПП РФ, приказы, СНиПы, ГОСТы, ПДД)

7. ССЫЛКИ НА ДРУГИЕ ДОКУМЕНТЫ
   - Какие документы упоминаются (номер, дата, тип)
   - Есть ли эти документы в деле (сравни с контекстом)

8. ЦИТАТЫ
   - Ключевые формулировки дословно (условия договора, выводы эксперта,
     признания сторон, резолютивная часть судебных актов)

9. КАЧЕСТВО OCR
   - Если текст плохо читается — указать
   - Если нечитаемые фрагменты — какие данные могли быть потеряны

Верни JSON по схеме.
```

### Адаптация промпта по типу документа

После определения типа документа (первые строки OCR-текста обычно достаточно) промпт дополняется специфическими указаниями. Но база одна — извлечение, не оценка.

#### Дополнение для искового заявления / встречного иска

```
ДОПОЛНИТЕЛЬНО ДЛЯ ИСКОВОГО:
- Каждое требование отдельно: предмет, сумма, правовое основание
- Были ли изменения/уточнения требований (увеличение, уменьшение, отказ от части)
- Перечень приложенных доказательств (из списка приложений к иску)
```

#### Дополнение для отзыва / возражений

```
ДОПОЛНИТЕЛЬНО ДЛЯ ОТЗЫВА:
- По каждому требованию истца: признаёт / оспаривает / молчит
- Каждый контрдовод отдельно — дословная формулировка
- Ходатайства: о чём просит (экспертиза, истребование, отложение, ст. 333 ГК)
- Заявления: исковая давность, ненадлежащий ответчик, подсудность
```

#### Дополнение для договора

```
ДОПОЛНИТЕЛЬНО ДЛЯ ДОГОВОРА:
- Все существенные условия: предмет, цена, сроки
- Порядок приёмки (как принимаются работы/товар)
- Ответственность: неустойка (ставка, лимит), штрафы
- Особые условия: подсудность, претензионный порядок (срок), форс-мажор
- Кто подписал: ФИО, должность, основание полномочий
- Дословные цитаты спорных/важных пунктов
```

#### Дополнение для экспертизы / заключения специалиста

```
ДОПОЛНИТЕЛЬНО ДЛЯ ЭКСПЕРТИЗЫ:
- Тип: судебная экспертиза или заключение специалиста (привлечён стороной)
- Эксперт: ФИО, квалификация, стаж, организация
- Предупреждение об уголовной ответственности (ст. 307 УК): есть подпись?
- Каждый вопрос и ответ — дословно
- Категоричный или вероятностный вывод
- Методы исследования, нормативы (СНиП, ГОСТ)
- Кто заявил ходатайство, кто оплатил, стоимость
```

#### Дополнение для платёжных документов

```
ДОПОЛНИТЕЛЬНО ДЛЯ ПЛАТЁЖНЫХ:
- Дата, плательщик, получатель, сумма
- Назначение платежа — ДОСЛОВНО (критично для привязки к договору)
- Для расписки: рукописная или печатная, есть ли паспортные данные,
  указан ли срок возврата, указаны ли проценты
```

#### Дополнение для судебного акта

```
ДОПОЛНИТЕЛЬНО ДЛЯ СУДЕБНОГО АКТА:
- Тип: определение / решение / апелляционное / кассационное
- Суд, дата, номер дела
- Резолютивная часть — ДОСЛОВНО
- Установленные факты — ДОСЛОВНО (потенциальная преюдиция)
- Вступил ли в силу
- Какие стороны совпадают с текущим делом
```

#### Дополнение для переписки / претензий

```
ДОПОЛНИТЕЛЬНО ДЛЯ ПЕРЕПИСКИ:
- Дата отправки и дата получения (могут отличаться)
- Доказательство отправки: почтовая квитанция, email, курьер
- Доказательство получения: уведомление, отслеживание
- Любые ПРИЗНАНИЯ в тексте — дословно ("согласны оплатить", "признаём задолженность")
- Для электронной переписки: нотариально заверена?
```

#### Дополнение для расчёта

```
ДОПОЛНИТЕЛЬНО ДЛЯ РАСЧЁТА:
- Формула: база × ставка × период = результат
- ПРОВЕРЬ АРИФМЕТИКУ. Если не сходится — укажи расхождение
- Период: дата начала, дата окончания, количество дней
- Для процентов по ст. 395: какие ключевые ставки ЦБ использованы и за какие периоды
```

### Обновление summary

После обработки каждого нового документа summary пересчитывается. DeepSeek получает:
- Текущий summary (JSON)
- Извлечение из нового документа (JSON)
- Инструкцию: обнови summary, добавив новую информацию

```
Ты — помощник судьи. Обнови сводку дела на основании нового обработанного документа.

ТЕКУЩАЯ СВОДКА:
{current_summary}

НОВЫЙ ДОКУМЕНТ ({doc_type}, {filename}):
{extracted_data}

ПРАВИЛА ОБНОВЛЕНИЯ:
1. Добавь новые факты, не дублируя существующие
2. Если новый документ уточняет данные — обнови (например, ИНН стороны)
3. Если новый документ противоречит существующим данным — добавь в contradictions
4. Добавь новые события в timeline (сохраняй хронологический порядок)
5. Добавь новые доводы сторон в arguments_plaintiff / arguments_defendant
6. Если документ упоминает другие документы, которых нет в деле — добавь в missing_documents
7. Обнови key_evidence если документ является важным доказательством
8. Обнови legal_norms если упоминаются новые нормы

Верни обновлённый summary в JSON.
```

### Хранение в БД

case_context хранится как JSONB-поле в таблице `cases`:

```sql
ALTER TABLE cases ADD COLUMN case_context JSONB DEFAULT '{}';
```

Размер: ~50–100 KB для крупного дела (40 документов). JSONB позволяет индексы и частичное чтение.

## 5. Модуль 3: Generate — генерация решения

### Вход

`case_context.summary` — не сырые тексты документов, а уже извлечённая и сведённая информация.

Размер summary для 40 документов: ~15–25K токенов. Комфортно влезает в контекст DeepSeek (64K) с запасом на промпт (~5K) и генерацию (~8K).

### Параметры

- Модель: DeepSeek V3 (deepseek-chat)
- max_tokens: 8192
- temperature: 0.3
- SSE-стриминг через существующий `deepseek.chat_stream()`

### Промпт генерации решения

```
Ты — помощник федерального судьи. Составь ПРОЕКТ судебного решения.

МАТЕРИАЛЫ ДЕЛА (сводка):
{case_summary}

УКАЗАНИЯ СУДЬИ:
{user_instructions}

НОРМЫ ПРАВА ИЗ БАЗЫ:
{matched_norms}

СТРУКТУРА РЕШЕНИЯ (ст. 198 ГПК РФ):

═══════════════════════════════════════════
ВВОДНАЯ ЧАСТЬ
═══════════════════════════════════════════
- Дата и место принятия решения
- Наименование суда
- Стороны (из summary.parties — полные данные)
- Представители
- Третьи лица
- Предмет спора

═══════════════════════════════════════════
ОПИСАТЕЛЬНАЯ ЧАСТЬ
═══════════════════════════════════════════
- Требования истца (из summary.claims — все, включая изменения)
- Позиция ответчика (из summary.arguments_defendant)
- Позиции третьих лиц
- Если есть встречный иск — требования по нему

═══════════════════════════════════════════
МОТИВИРОВОЧНАЯ ЧАСТЬ
═══════════════════════════════════════════

ОБЯЗАТЕЛЬНЫЕ ЭЛЕМЕНТЫ:

1. УСТАНОВЛЕННЫЕ ОБСТОЯТЕЛЬСТВА
   На основании summary.timeline и summary.key_evidence — что суд считает установленным.
   Каждый факт — со ссылкой на доказательство.

2. ОЦЕНКА ДОКАЗАТЕЛЬСТВ
   На основании summary.key_evidence и summary.contradictions — какие доказательства
   приняты, какие отклонены и ПОЧЕМУ.

   Если есть противоречия (summary.contradictions) — мотивировать выбор:
   почему суд доверяет одному доказательству больше, чем другому.

3. ВСЕ ДОВОДЫ СТОРОН
   По КАЖДОМУ доводу из summary.arguments_plaintiff и summary.arguments_defendant
   суд ОБЯЗАН высказаться.

   ⚠️ Если довод отклоняется — объяснить почему.
   Пропуск довода = основание для отмены (п. 2 ч. 1 ст. 330 ГПК).

4. ПРИЗНАННЫЕ ФАКТЫ
   Из summary.admissions — признанные обстоятельства не нуждаются в доказывании (ст. 68 ГПК).

5. ПРИМЕНЁННЫЕ НОРМЫ
   Ссылки на нормы из summary.legal_norms + matched_norms из базы.
   Маркеры [[NORM:uuid]] для замены на человекочитаемые ссылки.

   Суд определяет правоотношения САМОСТОЯТЕЛЬНО, даже если стороны
   неправильно квалифицировали спор (п. 9 ППВС №25).

6. РАСЧЁТЫ
   Если в summary есть суммы — проверить и привести итоговый расчёт.
   Если стороны оспаривают расчёт — мотивировать.

═══════════════════════════════════════════
РЕЗОЛЮТИВНАЯ ЧАСТЬ
═══════════════════════════════════════════
- Удовлетворить / отказать (полностью / частично) — с ТОЧНЫМИ суммами
- Распределение судебных расходов ПРОПОРЦИОНАЛЬНО удовлетворённым требованиям
- Госпошлина: с кого, сколько
- Расходы на представителя: разумные пределы
- Обеспечительные меры: отменить / сохранить (если были)
- Немедленное исполнение (ст. 211 ГПК): если применимо
- Порядок обжалования: в какой суд, в какой срок

ФОРМАТ:
- Маркеры норм: [[NORM:uuid]]
- Юридический язык, деловой стиль
- Суммы: цифрами и прописью
```

### Категорийные дополнения к промпту

Если из summary определяется категория дела, к промпту добавляются дополнительные инструкции. Юридическая оценка — здесь, а не на этапе извлечения.

#### Потребительские споры (ЗоЗПП)

```
ПОТРЕБИТЕЛЬСКИЙ СПОР — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- Штраф 50% от присуждённой суммы (п. 6 ст. 13 ЗоЗПП) — взыскивается СУДОМ,
  даже если истец не заявлял
- Моральный вред — презюмируется при нарушении прав потребителя (ст. 15 ЗоЗПП)
- Неустойка: 1%/день (товар, ст. 23 ЗоЗПП) / 3%/день (услуга, ст. 28 ЗоЗПП)
- Бремя доказывания: на продавце/исполнителе
- Госпошлина: потребитель освобождён до 1 000 000 руб.
```

#### Трудовые споры

```
ТРУДОВОЙ СПОР — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- Бремя доказывания законности увольнения — на РАБОТОДАТЕЛЕ
- Процедура увольнения: каждый шаг (уведомление, объяснительная, сроки, приказ)
- Восстановление на работе: немедленное исполнение (ст. 211 ГПК)
- Средний заработок за вынужденный прогул
- Моральный вред: обязателен при ЛЮБОМ нарушении трудовых прав (ст. 237 ТК)
- Компенсация за неиспользованный отпуск
- Срок обращения: 1 месяц (увольнение), 1 год (зарплата) — ст. 392 ТК
```

#### Семейные споры

```
СЕМЕЙНЫЙ СПОР — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- Интересы ребёнка — ПРИОРИТЕТ №1 (ст. 65 СК)
- Заключение органа опеки (ст. 78 СК) — без него решение отменят
- Мнение ребёнка старше 10 лет (ст. 57 СК)
- Алименты: 1/4 на одного, 1/3 на двоих, 1/2 на трёх+ (ст. 81 СК)
- Немедленное исполнение: алименты (ст. 211 ГПК)
- Раздел имущества: равные доли по умолчанию (ст. 39 СК)
```

#### Наследственные споры

```
НАСЛЕДСТВЕННЫЙ СПОР — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- Круг наследников, очерёдность (ст. 1142–1145 ГК)
- Обязательная доля (ст. 1149 ГК): не менее 1/2 законной доли
- Способ принятия: нотариальное или фактическое
- Срок: 6 месяцев (ст. 1154 ГК)
- Долги наследодателя: в пределах стоимости наследства
```

#### ДТП и страхование

```
ДТП / СТРАХОВАНИЕ — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- ОСАГО лимиты: 400 000 (имущество), 500 000 (жизнь/здоровье)
- Износ: с учётом износа (ОСАГО) vs без износа (к виновнику)
- УТС: взыскивается с виновника
- Источник повышенной опасности (ст. 1079 ГК): ответственность БЕЗ вины
- Суброгация (ст. 965 ГК)
```

#### ЖКХ / жилищные споры

```
ЖКХ — ДОПОЛНИТЕЛЬНО ПРОВЕРЬ:
- Тарифы: утверждены муниципалитетом?
- ПП РФ №354 (правила предоставления коммунальных услуг)
- Акт осмотра / акт о заливе: кем составлен, когда
- Кворум общего собрания собственников
- Срок оспаривания решений ОСС: 6 месяцев (ст. 46 ЖК)
```

## 7. Edge cases

### 7.1. Дубликаты

Судья загрузил одну фотографию дважды.
- **Решение:** SHA-256 хеш OCR-текста. Если совпадает — пропускаем.
- **UX:** «Этот документ уже загружен (совпадает с IMG_001.jpg)»

### 7.2. Нечитаемый скан

OCR вернул мусор или пустой текст.
- **Решение:** если DeepSeek не смог извлечь ни одного поля — помечаем как нечитаемый.
- **UX:** «Документ IMG_015.jpg не удалось распознать. Попробуйте загрузить фото лучшего качества.»
- Документ НЕ добавляется в контекст.

### 7.3. Удаление документа

Судья хочет удалить загруженный документ.
- **Решение:** кнопка «Удалить» рядом с каждым документом. При удалении — пересчитать summary без этого документа (один вызов DeepSeek ~$0.01).
- Альтернатива: полный пересчёт summary из оставшихся extracted — без LLM, просто пересборка.

### 7.4. Доработка решения

Судья получил решение, хочет поправить.
- Контекст уже собран — перезапускаем Generate с тем же summary + комментарии судьи.
- Стоимость: ~$0.02. Время: ~30 секунд.

### 7.5. Очень длинный документ (договор на 50 страниц)

Один документ > 64K токенов контекста DeepSeek.
- **Решение:** разбить на части (по ~40 страниц), обработать каждую часть как отдельный документ с пометкой «часть 1 из N документа [название]».
- Summary объединит части автоматически.

### 7.6. Summary переполняет контекст Generate

Summary > 30K токенов (маловероятно при 40 документах, возможно при 100+).
- **Решение:** сжатие summary — убрать дубли, сократить цитаты, оставить только ключевые факты.
- Отдельный вызов DeepSeek: «Сожми эту сводку до 20K токенов, сохрани все доводы сторон, суммы и хронологию.»

## 10. Порядок реализации

### Фаза 1: Ядро (MVP)

1. Миграция: добавить `case_context` JSONB в таблицу `cases`
2. `case_context.py` — CRUD для контекста дела
3. `extract_prompts.py` — базовый промпт + 3 дополнения (исковое, отзыв, договор)
4. `ingest.py` — OCR → извлечение → обновление summary
5. `generate.py` — генерация решения из summary
6. Эндпоинт upload_photos (SSE) в `cases.py`
7. Тесты

### Фаза 2: Полнота

8. Остальные дополнения к промптам (экспертиза, платёжки, переписка, расчёт, судебный акт)
9. Категорийные дополнения к Generate (ЗоЗПП, трудовые, семейные)
10. Удаление документа + пересчёт summary
11. Обработка длинных документов (разбивка)

### Фаза 3: Оптимизация

12. Сжатие summary при переполнении
13. Кеш: не пересчитывать summary если документы не изменились
14. Frontend: отображение списка обработанных документов, прогресс-бар
